# 2026-07-05 工作日志

## iOS Safari 打卡确认按钮无响应修复
- 问题：macOS Safari 可正常打卡，但 iOS Safari 点击「完成打卡」无反应，「取消」正常。
- 根因：弹窗用外层 div `onClick={onClose}` + 内层 div `onClick={e.stopPropagation()}` 防穿透。iOS Safari 上 stopPropagation 不可靠：点击内部按钮时 click 事件会"泄漏"到外层遮罩，导致 onConfirm 和 onCancel 同时触发，onCancel 立即关闭弹窗使组件卸载，onConfirm 的异步操作被中断。「取消」不受影响因为它和 onCancel 是同一动作（幂等）。
- 修复：改用 `if (e.target === e.currentTarget) onClose()` 模式——仅在点击遮罩本身时关闭，点击任何子元素都不触发关闭。移除 stopPropagation。遮罩加 `cursor: pointer`、内容区加 `cursor: default` 确保 iOS 正确识别可点击区域。
- 同步修复 RedemptionPanel（相同模式）。
- 构建测试通过，服务已重启。

## iOS Safari 打卡修复（第三次，找到真根因）
- 真根因：`crypto.randomUUID()` **仅在安全上下文（HTTPS / localhost）可用**。
  iPad 通过局域网 HTTP IP（如 http://192.168.1.x:3000）访问时属于非安全上下文，
  iOS Safari 上 `crypto.randomUUID` 为 `undefined`。
- 点击「完成打卡」时 `generateRequestId()` 调 `crypto.randomUUID()` 直接抛错，
  mutation 失败，onConfirm 无 try-catch → 静默失败 → 弹窗不关、无反馈 = "没反应"。
  「取消」不涉及 requestId 生成，所以正常。macOS 上 localhost 是安全上下文，所以正常。
- 修复：`packages/shared/src/logic.ts` 的 `uuid()` 增加 fallback——
  `crypto.randomUUID` 可用时用它，否则用 `crypto.getRandomValues()`（所有上下文可用）
  手动构造 UUID v4，最终兜底 Math.random。
- 前两次的弹窗结构重构（兄弟元素、touch-action）虽非此 bug 的解法，但作为 iOS 健壮性改进保留。
- 构建测试通过，服务已重启。
